## ⚡ 1. The Core Definition & Trigger
*   **Two-Sentence Trigger:** A Fencing Token is a monotonically increasing sequence number generated by a lock server every time a distributed lock is acquired. The database validates this token during writes, rejecting any request containing a token lower than the highest token it has already processed, thereby protecting the system against delayed writes from clients whose locks have expired.
*   **Scalability Dimension:** Primary: **Data Consistency (Write Integrity)** & **Race Condition Protection**.

---

## ⚖️ 2. Trade-offs & Deep Dive

### The Lock-GC Pause Race Condition
1.  **Lock Acquisition:** Client 1 acquires a distributed lock and receives a lease.
2.  **GC Pause:** Client 1 undergoes a Stop-the-World garbage collection pause.
3.  **Lease Expiry:** While Client 1 is paused, its lock lease expires.
4.  **New Acquisition:** Client 2 acquires the same lock.
5.  **Concurrent Writes:** Client 2 writes data to the DB. Client 1 wakes up from its GC pause, assumes its lock is still valid, and writes its data, overwriting and corrupting Client 2's updates.

```
Client 1 ──► [Acquire Lock] ──► Token: 33 ──► [GC PAUSE]
                                                  │
Client 2 ──► [Acquire Lock] ──► Token: 34 ──► [Writes to DB (Accepts 34)]
                                                  │
Client 1 ──► [Wakes Up] ─────► [Writes to DB (Token 33 < 34 -> REJECTED)]
```

### The Fencing Solution
By passing the token (e.g., 33 vs 34) down to the storage tier, the storage engine acts as the gatekeeper. Because the DB has already committed a transaction with token 34, it immediately rejects Client 1's request containing token 33.

*   **Pros:** 100% reliable protection against split-brain writes and network pause delays.
*   **Cons:** Requires database support (the database must track the highest processed token per record).

---

## 💥 3. Resiliency & Operations
*   **Observability (The "Signal"):**
    *   `Rejected write rates due to stale fencing tokens`: Indicates high GC pauses or network routing delays in client nodes.
*   **Blast Radius (The "Impact"):**
    *   Low. Stale writes are safely isolated and blocked, preventing database contamination, though clients will receive transaction write failures.

## 🚫 4. Interview Playbook

### Common Mistakes (The "Junior" Signals)
*   Assuming distributed locks are 100% safe without fencing tokens (Martin Kleppmann proved that locks are unsafe without fencing due to GC pauses and network delays).
*   Not knowing that tokens must be monotonically increasing.

### Interview Tip (The "Strong Hire" Signal)
> *"Distributed locks are vulnerable to client garbage collection pauses. To secure our write transactions, we implement **Fencing Tokens**. Our ZooKeeper lock coordinator issues an increasing sequence ID with the lock, and our PostgreSQL database checks this token before updating rows, ignoring late writes."*
